# Entity Relationship Diagram

## Entities & Their Core Attributes

1. User: userId, name, email, passwordHash, phone, role, familyDetails, vehicleInfo, status (Active | Inactive), createdAt
2. Unit: unitId, unitNumber, unitType, floor, monthlyCharges, status
3. OwnershipRecord: recordId, unitId, userId, ownershipType, startDate, endDate
4. Bill: billId, unitId, generatedBy, billType, amount, dueDate, status
5. Payment: paymentId, billId, paidBy, amountPaid, paymentMethod, transactionRef, paidAt
6. Expense: expenseId, recordedBy, category, amount, date, description
7. FinancialReport: reportId, generatedBy, reportType (Income | Expense | BalanceSheet | Defaulters), dateRangeStart, dateRangeEnd
8. Notice: noticeId, postedBy, title, description, priority, postedAt
9. Complaint: complaintId, submittedBy, unitId, category, description, status (Pending | In Progress | Resolved), ticketId
10. SOSAlert: alertId, triggeredBy, locationInfo, status, emergencyContacts (JSON)
11. SOSResponse: responseId, alertId, guardId, acknowledgedAt
12. Poll: pollId, createdBy, question, options, startDate, endDate, status
13. Vote: voteId, pollId, votedBy, selectedOption
14. Visitor: visitorId, name, phone, idProofType, idProofNumber
15. VisitorLog: logId, visitorId, unitId, loggedBy, entryTime, exitTime, purpose, approvalId
16. GuestApproval: approvalId, approvedBy, visitorId, unitId, validDate, status
17. Staff: staffId, name, role (Maid | Driver | Vendor | Other), phone, assignedUnitId
18. StaffAttendance: attendanceId, staffId, recordedBy, entryTime, exitTime, date
19. GateAccessLog: accessId, managedBy, entityType, entityId, action, timestamp
20. PatrolLog: patrolId, guardId, routeId, timestamp
21. Facility: facilityId, name, type, capacity, status
22. FacilityBooking: bookingId, facilityId, bookedBy, date, timeSlotStart, timeSlotEnd, status
23. Inventory: itemId, managedBy, name, category, quantity, condition, purchaseDate, lastUpdated

## Relationships with Cardinality

### Member & Property Management

| Relationship  | Entities                      | Cardinality | Notes |
|---------------|-------------------------------|-------------|-------|
| Resides In    | Resident -> Unit              | M : 1       | Multiple residents (owner + tenants) can be linked to one unit. |
| Has History   | Resident -> OwnershipRecord   | 1 : M       | The OwnershipRecord is the authoritative source for current residency and history. A resident can have history across multiple units over time. |
| Belongs To    | OwnershipRecord -> Unit       | M : 1       | Many ownership records (history) per unit; one active at a time. |

### Financial Management

| Relationship | Entities                              | Cardinality | Notes |
|--------------|---------------------------------------|-------------|-------|
| Has Bills    | Unit -> Bill                          | 1 : M       | One unit accumulates many recurring bills. |
| Generated By | Bill -> User (Admin/Accountant)       | M : 1       | Many bills generated by one admin/accountant. |
| Settled By   | Bill -> Payment                       | 1 : 1       | Each bill has at most one payment record. |
| Makes        | Resident -> Payment                   | 1 : M       | A resident makes many payments over time. |
| Records      | User (Admin/Accountant) -> Expense    | 1 : M       | One user logs many expense entries. |
| Generates    | User (Admin/Accountant) -> FinancialReport | 1 : M  | Multiple reports generated by one authorized user. |
| Aggregates   | FinancialReport <- Bill + Expense     | Derived     | Reports draw data from both Bills and Expenses (logical dependency, not a hard FK). |

### Communication & Collaboration

| Relationship | Entities                                    | Cardinality | Notes |
|--------------|---------------------------------------------|-------------|-------|
| Posts        | User (Admin) -> Notice                      | 1 : M       | Admin posts many notices. |
| Submits      | Resident -> Complaint                       | 1 : M       | One resident submits many complaints over time. |
| Concerning   | Complaint -> Unit                           | M : 1       | Many complaints can be tied to the same unit. |
| Triggers     | Resident -> SOSAlert                        | 1 : M       | A resident can trigger multiple alerts. |
| Notifies     | SOSAlert -> SecurityGuard (via SOSResponse) | M : M       | One alert can notify multiple guards; one guard receives many alerts — resolved through SOSResponse. |
| Creates      | User (Admin) -> Poll                        | 1 : M       | Admin creates many polls. |
| Casts        | Resident -> Vote                            | 1 : M       | A resident casts one vote per poll (enforced by unique constraint on pollId + votedBy). |
| Has          | Poll -> Vote                                | 1 : M       | One poll holds many votes. |

### Security & Visitor Management

| Relationship   | Entities                        | Cardinality | Notes |
|----------------|---------------------------------|-------------|-------|
| Has Logs       | Visitor -> VisitorLog           | 1 : M       | Same visitor can visit multiple times; each visit is a separate log. |
| Visiting       | VisitorLog -> Unit              | M : 1       | Many visitor logs can reference the same unit. |
| Logged By      | VisitorLog -> SecurityGuard     | M : 1       | One guard logs many visitor entries/exits. |
| Pre-Approves   | Resident -> GuestApproval       | 1 : M       | A resident can pre-approve many guests. |
| For            | GuestApproval -> Visitor        | M : 1       | Multiple approvals can exist for the same visitor over different dates. |
| Links To       | GuestApproval -> VisitorLog     | 1 : 1       | When a pre-approved guest arrives, their approval is linked to the actual log entry (nullable until arrival). |
| Has Attendance | Staff -> StaffAttendance        | 1 : M       | Each staff member has daily attendance records. |
| Records        | SecurityGuard -> StaffAttendance| 1 : M       | One guard records attendance for many staff. |
| Manages        | SecurityGuard -> GateAccessLog  | 1 : M       | One guard manages many gate access events. |

### Amenity & Inventory Management

| Relationship | Entities                      | Cardinality | Notes |
|--------------|-------------------------------|-------------|-------|
| Books        | Resident -> FacilityBooking   | 1 : M       | One resident makes many bookings. |
| Is Booked    | Facility -> FacilityBooking   | 1 : M       | One facility has many bookings across time slots. |
| Manages      | User (Admin) -> Inventory     | 1 : M       | One admin manages many inventory items. |

## Key Constraints & Business Rules to Enforce

- Unique vote per poll: (pollId, votedBy) must be unique in the Vote table — a resident cannot vote twice on the same poll.
- No double booking: Ensure (facilityId, date, timeSlotStart, timeSlotEnd) do not overlap.
- Bill-Payment integrity: A Payment cannot exist without a corresponding Bill; a Bill's status must update to "Paid" upon Payment insertion.
- Guest approval nullable link: VisitorLog.approvalId is nullable — walk-in visitors have no pre-approval; pre-approved guests get the FK filled when they arrive.
- Role enforcement: Only Admin can post notices. Only Residents can submit complaints, trigger SOS, and book facilities.
- SOS Workflow: System automatically notifies security guards and designated external contacts upon Resident trigger.
